Most "prompt engineering" content is written for people building demos. Here's what I do when the output is going into a real codebase.

## Give it the constraints, not the solution

Bad: "Write a login function using bcrypt with a 12-char password."
Good: "Write a login function for a Node/Express API. Passwords hashed with bcrypt cost 12. Return typed errors for invalid credentials, locked account, and MFA required. Don't leak which one it was."

## Include the file's existing style

Paste 20 lines of surrounding code before asking. The model matches whatever conventions it sees.

## Ask for the test first

"Write a Vitest suite for this function. Cover the happy path, empty input, and one adversarial input." Then have it implement the function. Reverses the usual failure mode where the code passes tests only the model wrote to fit it.

## Force it to explain edge cases before writing

"Before you write any code, list five edge cases this function must handle." Cheap step, dramatically better output.

## Reject the first answer that looks fine but you don't fully understand

If you can't explain it, you can't maintain it. Ask again with your specific confusion.

Prompting well is basically technical writing. The clearer your spec, the better the code.
