App Router confused me for months. Here's the mental model that finally made it click.

## Everything is a server component by default

That's the whole shift. You're writing Node code that renders to HTML. `useState` doesn't work here — not because Next.js is broken, but because there's no client yet. Add `"use client"` when you actually need interactivity.

## Layouts persist. Pages don't.

`layout.tsx` renders once and stays mounted across navigation. `page.tsx` re-renders per route. Put nav, sidebars, and providers in layouts. Put content in pages.

## Data fetches inline

No more `getServerSideProps`. Just `await fetch()` inside the server component. Deduplication is automatic across the tree.

## The client boundary is a leaf, not a slice

Once a file is `"use client"`, everything it imports gets bundled for the client too. Keep client components small and push them to the leaves.

## Server actions replace half your API routes

Form submissions, mutations, simple CRUD — server actions handle it with less code and better types. Keep route handlers for public APIs and webhooks.

Get these five things right and App Router stops fighting you.
